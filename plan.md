- [x] Reworked enemy patrol in `boss.gd` (shared by all 5 enemy/boss scenes -- CheeseEnemy, JalapenoEnemy, SalsaBowlEnemy, HotSauceBoss, QuesoGrandeBoss) after observing enemies getting stuck pushing against the new jump obstacles: patrol was purely distance-based (`patrol_half_width` from spawn point) with zero awareness of obstacles or platform edges, so `move_and_slide()` would just silently stop the enemy dead against a wall forever. Enemies now also turn around immediately on `get_slide_collision_count() > 0` (hit something solid), and before walking off a platform edge via a short raycast checking for floor ahead (`PhysicsRayQueryParameters2D`, no new scene nodes needed since it queries the physics space directly from script). No scene file changes required -- picked up by all 5 automatically.
- [x] Window/background parallax went through several wrong turns before landing on the right setup -- documenting the actual final state and the costly detour, since it's a real lesson (see `feedback_verify_engine_defaults_before_relying.md` in AI memory too):
  1. First redesign dropped the flat `SkyTop`/`SkyMid`/`SkyHorizon` bands entirely and tiled the window art itself as the whole background (per user direction: "have the whole window image be the entire background and have it parallax scroll").
  2. Misdiagnosed the window as stuck in `CanvasLayer` screen-space (it wasn't -- the user confirmed the original `ParallaxBackground` setup "was parallaxing just fine") and replaced working code with a custom hand-rolled parallax script -- wasted effort chasing a bug that didn't exist.
  3. Correctly reverted to Godot's native `ParallaxBackground`/`ParallaxLayer` on the user's direct suggestion ("wouldn't it be smarter to use godot's own APIs") rather than keep patching the custom script. The real, valid fix this surfaced: `ParallaxLayer.motion_scale` is a `Vector2`, so `Vector2(0.4, 1.0)` gives horizontal parallax depth while `Y=1.0` keeps full vertical lockstep with the camera -- necessary because any Y-axis parallax reduction tears a gap open between the background and the ground on every jump (the ground is normal un-parallaxed world content; the background must track camera Y 1:1 to stay glued to it).
  4. Attempted a plain top-level `ColorRect` "backdrop" (behind the tiled window art, to hide gray showing past tile edges/seams) -- wrong, since `ParallaxBackground` is specifically designed to always composite as a background layer regardless of tree position, so a plain scene-tree node actually draws *in front of* it, hiding the window art. Correct fix: the backdrop must live *inside* the `ParallaxBackground`, as its own `ParallaxLayer` (Level1 uses a separate layer with `motion_scale (0,0)`, screen-locked so it always fully covers the viewport regardless of camera range; Level2 puts it directly in the window layer instead, scrolling with the windows at the 0.4 rate, sized generously to still cover the visible range).
- [x] User then made a round of manual fixes directly in the Godot editor (documented here since the AI didn't make these, just committed them as instructed -- "I have altered the game manually in Godot. Do not revert any changes."): ground visuals and hazard-fill textures extended taller (larger `offset_bottom`) to push their bottom edge further down and reduce remaining gray below the ground line; L1 string lights repositioned higher (`y -20` -> `y -305`) to sit correctly against the taller backdrop; `Player.tscn`'s `Camera2D` given `offset = Vector2(150, -225)` to shift the framing; L2's `Chip1-4` individually repositioned/rescaled by hand; both levels' player spawn position nudged slightly; `GroundA`'s collision widened and offset on both levels.
- [x] Resolved the Level1 node-structure question flagged last session: the nested `WindowParallax/WindowLayer/Window1` child (offset ~-1312 units) is a deliberate user-made fix, added by hand in the editor to extend the background's repeat coverage further left -- not an accident, left as-is. `GroundA`'s `CollisionShape2D` turned out already clean (single shape, no nesting) on inspection -- that half of the earlier note was stale. `JumpObstacleSmall1` did have a genuine duplicate nested `CollisionShape2D` (two identical shapes, one parented inside the other); removed the redundant inner one.
- [x] Found and fixed a real bug via first playtest: enemies/bosses (`boss.gd`, shared by all 5) were vibrating in place instead of patrolling -- `if get_slide_collision_count() > 0: patrol_direction = -patrol_direction` (added last session to turn around on hitting a wall) can't tell a wall hit apart from simply standing on flat ground, since Godot's `CharacterBody2D` floor-snap contact registers as a slide collision every single physics frame too (`normal = (0, -1)`). Confirmed with a headless repro before *and* after the fix rather than assuming: flipping direction every frame nets ~zero movement, reading as a fast side-to-side jitter -- exactly what showed up in-game. Fix: only flip on a near-horizontal collision normal (`abs(normal.x) > 0.5`), which a repro run over 60 frames of pure floor contact confirmed never false-triggers, while still catching genuine wall hits by definition (an axis-aligned wall's normal is horizontal). Also widened `QuesoGrandeBoss`'s `patrol_half_width` from 20 to 60 (matching Hot Sauce's) -- at 20 it was actually *smaller* than the boss's own collision half-width (22), leaving almost no real room to patrol even after the jitter fix.
